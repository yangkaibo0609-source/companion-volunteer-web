# Token 与执行效率约束

本轮请优先节省上下文、推理和输出 token，同时保证修改正确。

## 1. 不要重复读取无关内容

开始前只读取：

* 根目录 `AGENTS.md`
* 项目设计 Skill 或 `DESIGN_SYSTEM.md`
* 当前任务直接涉及的组件、数据文件和样式文件
* `访谈故事.docx`
* `封面.zip`

使用 `rg`、文件名搜索和 import 关系定位代码。

禁止：

* 无目的遍历整个仓库；
* 全量读取 `node_modules`、`dist`、构建产物；
* 重复读取已经确认且未发生变化的文件；
* 搜索与本任务无关的页面；
* 联网查找新设计案例；
* 重新分析已经确定的视觉方案。

## 2. 不要重复解释需求

不要在回复中复述完整任务。

开始执行前，只输出不超过 6 行的简短计划，说明：

1. 定位哪些文件；
2. 插入哪些封面；
3. 更新哪些黑板文案；
4. 如何验证。

随后直接执行。

## 3. 限定修改范围

优先只修改：

* 页面流程或路由入口；
* 数据黑板相关组件；
* 黑板文案数据文件；
* 章节封面组件；
* 封面图片资源；
* 必要的 CSS；
* 与流程连接直接相关的 Summary 组件。

除非编译或功能确实需要，不要：

* 重构无关组件；
* 重命名大量文件；
* 改动访谈票根内部实现；
* 改动文字游戏逻辑；
* 安装新依赖；
* 更换动画库；
* 重写全局状态管理；
* 格式化整个项目。

## 4. 复用现有实现

优先复用现有：

* 黑板组件；
* 黑板展开交互；
* 图表按需加载逻辑；
* 水彩背景；
* 字体系统；
* 访谈票根系统；
* IntersectionObserver 或现有滚动动画；
* 图片懒加载工具。

不要为了同一功能创建第二套组件。

只有现有组件无法满足需求时，才新增最小组件。

## 5. 文案处理方式

不要把《访谈故事.docx》全文复制到过程输出中。

只需：

1. 从文稿中定位指定段落；
2. 提取原句；
3. 写入 `dataBlackboardContent.ts`；
4. 保留来源章节字段供开发核对；
5. 不在聊天回复中重新输出所有正文。

文稿内容只能删减和拆段，禁止重新改写。

发现数字冲突时：

* 不要长篇分析；
* 记录到 `DATA_CONFLICTS.md`；
* 页面暂时使用本轮明确指定的文稿版本；
* 最终回复用一行列出冲突。

## 6. 素材处理方式

只解压并处理 `封面.zip` 中的 5 张图片。

不要搜索其他图片，不要生成新图片。

图片转换时：

* 每张只生成必要的 WebP/AVIF 尺寸；
* 不生成大量重复规格；
* 总封面 eager；
* 其余封面 lazy；
* 保留原图；
* 不重复压缩已经存在的合适版本。

## 7. 验证策略

不要在每改一个文件后都运行完整构建。

按以下顺序验证一次：

1. TypeScript/静态检查；
2. `npm run lint`；
3. `npm run build`。

如果失败：

* 只读取报错涉及的文件；
* 只修复相关问题；
* 再运行对应检查；
* 不重新检查整个设计方案。

除非用户明确要求，不生成额外截图、测试报告或性能报告。

## 8. 输出限制

最终回复控制在 15 行以内，只包括：

* 修改文件；
* 5 张封面插入位置；
* 文稿是否按原文删减；
* 是否发现数据冲突；
* 是否保留黑板和票根交互；
* lint 结果；
* build 结果；
* 尚未解决的问题。

不要：

* 重复完整需求；
* 粘贴大段代码；
* 粘贴全部文案；
* 描述每一个 CSS 调整；
* 输出冗长设计说明。

## 9. 停止条件

完成本轮明确任务并通过验证后立即停止。

不要顺手继续：

* 美化其他页面；
* 增加新交互；
* 重构票根；
* 添加 Three.js；
* 添加声音；
* 优化未被要求的模块。

如遇关键素材缺失、数据冲突或无法确认的页面位置，停止猜测，简洁报告问题。
